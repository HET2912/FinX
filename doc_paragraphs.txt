316: STUDY OF CURRENT SYSTEM
317: Currently, many e-commerce platforms exist, offering features like product listings, online shopping, and order management. However, most traditional platforms either have cluttered user interfaces, limited administrative control, or lack a personalized and seamless shopping experience.
318: Existing systems like Amazon, Flipkart, and Myntra provide essential online shopping functionalities, but they often come with limitations such as:
319: • Limited administrative control over product management and order handling processes.• Overloaded interfaces that can be confusing for first-time users or customers.• Inefficient filtering and search mechanisms, making it harder for users to find relevant fashion products quickly.• Lack of personalization and engaging user experience in some cases.
320: FAB FASHION aims to address these issues by providing a clean, efficient, and user-friendly e-commerce solution with robust admin control, secure authentication, and a modern shopping experience for both customers and administrators.
321: WEAKNESS OF CURRENT SYSTEM
322: 1. Limited Admin Control
323: • Many platforms provide limited administrative control, restricting efficient management of products, users, and orders.• Lack of centralized tools to easily update product details, manage inventory, or handle order statuses
324: 2.Poor User Experience & Overloaded Interfaces
325: • Several e-commerce websites have cluttered layouts, making navigation confusing for users.• Interfaces are not always fully responsive, leading to inconsistent experiences across devices.
326: 3. Lack of Transparency and Product Insight
327: • Some platforms offer limited product details, reviews, or ratings.
328: • Customers often face difficulty in making informed purchase decisions due to insufficient information.
329: 4. Limited Customization & Flexibility
330: • Product management and filtering options are often rigid and not adaptable to different user preferences.• Dependence on external integrations for essential features like authentication, payments, or tracking.
331: 5. Ineffective Search and Filtering
332: • Users may struggle with irrelevant search results and limited filtering options.• Difficulty in quickly finding products based on category, price range, or preferences.

678: When a user registers with the application, their details are stored in the User table. When an administrator adds new products, the application stores the product details in the Product table.
679: When users browse or search for products, the application queries the database to retrieve relevant items from the Product table based on search criteria and filters. When a user adds items to the cart, the selected products are stored in the Cart table.
680: During checkout, the application retrieves cart and address details, processes the payment, and stores the order information in the Order table. Users can track their orders, while administrators can view and update order statuses, which are reflected in the database.
681: Additionally, users can provide reviews for purchased products, and these are stored in the Review table. Address details added by users are maintained in the Address table and used during order placement.

722: CHAPTER 7: TESTING
723: Testing is a process of executing a program with the intent of finding bugs that makes the application fail to meet the expected behaviour. Regardless of the development methodology, the goal of testing is to make sure that what is created does what it is supposed to do. Testing plays a critical role for assuring quality and reliability of the software. I have included testing as a part of development process. The test cases should be designed with maximum possibilities of finding the errors or bugs. Various level of testing are as follows.
724: Testing Levels
725: Unit testing: Unit testing tests the functionality of individual units of source code. It is the smallest component of a testable software that works in isolation with other parts of the code. I have done unit testing for various individual components of the source code to uncover errors within the boundary of the application.
726: Integration testing: Integration testing focuses on the design and construction of the software. Here the individual components that are tested using unit tests are combined and tested as a group. Its primary purpose is to expose the defects associated with the interfacing of modules. It checks if the modules perform the desired functionality when integrated together.
727: System testing: System testing is performed on a completely integrated system to see if it meets the requirements.
728: Regression testing: Regression testing aims at verifying the functionality of the software that is previously tested and to which changes are made. It is to ensure the old software still works with new changes.
729: Acceptance testing: Acceptance testing is conducted to verify if the system compliance the business requirements.
730: Adhering to the levels of testing, Unit testing is performed on individual components of the system ensuring the expected behaviour. Later, I have integrated various components together and performed Integration testing. Once the integration testing is done, I have performed System testing and ensured the application works as per the requirements. Finally, acceptance testing is performed to check if the client accepts the system.
